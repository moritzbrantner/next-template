# Agent Instructions

## Project Purpose

`next-template` is a Bun workspace Next.js 16 App Router scaffold. It combines localized routing, credential auth, account lifecycle flows, admin/reporting features, MDX content, Drizzle/Postgres foundations, and two local app-pack packages.

## Ownership Boundaries

- `app/`: canonical Next.js routes, layouts, pages, and route handlers.
- `src/`: domain, infrastructure, runtime, auth, data, and shared application logic used by the root app.
- `components/`: reusable app UI components. Keep feature-specific behavior close to the feature when possible.
- `messages/`, `i18n/`, and `apps/showcase/messages/`: locale catalogs and routing helpers. Keep locale keys in sync across supported locales.
- `apps/showcase/`: showcase app-pack manifest, MDX content roots, examples, and contract tests.
- `packages/app-pack/` and `packages/app-pack-react/`: local packages that publish built `dist` entrypoints. Source lives in `src/`; do not hand-edit built output.
- `drizzle/` and `drizzle/meta/`: checked-in database migrations and snapshots generated by Drizzle. Generate them through Drizzle commands rather than editing snapshots by hand.
- `docs/`: architecture, release, production-readiness, and contributor workflow notes.
- `public/local-profile-images/`, `.generated/`, `.next/`, `out/`, `dist/`, `coverage/`, `test-results/`, and `playwright-report/`: generated or local-only outputs. Do not edit or commit them.

## Commands

Use Bun `1.3.14`, matching `packageManager` and CI.

```bash
bun install --frozen-lockfile
bun run dev
bun run test
bun run lint
bun run format:check
bun run format
bun run build
bun run verify
```

- `bun run dev`: starts an ephemeral Postgres database, applies migrations, seeds baseline users, regenerates `db-schema.json`, and runs the app.
- `bun run dev:app`: runs `next dev` after package builds when long-lived local services are already running.
- `bun run test`: fastest meaningful test pass; currently aliases `bun run test:unit`.
- `bun run lint`: existing formatter check, `oxfmt --check .`.
- `bun run format:check`: checks formatting without mutation.
- `bun run format`: writes formatter changes with `oxfmt --write .`.
- `bun run build`: builds local packages and the production Next app.
- `bun run verify`: repo hygiene report plus the full CI confidence path from `checks:main`.
- `bun run checks:nightly`: app lint/typecheck/unit tests plus workspace package lint/typecheck/tests.
- `bun run checks:beta`: nightly checks plus integration tests.
- `bun run checks:main`: beta checks plus database check, production build, and e2e setup/tests.
- `bun run checks:all`: current full CI tier wrapper, equivalent to `checks:main`.
- `bun run hygiene`: reports dirty status, untracked files, upstream/ahead/behind state, generated directories, and local-only ignore coverage.

There is no root `release` or `publish` script. Package publishing is documented in `docs/releasing-packages.md` and should only be run from the relevant `packages/*` directory after package verification.

## Verification Guidance

- Before editing, run `git status --short --branch` and note unrelated user changes. This repository often has generated outputs locally, so avoid broad cleanup unless asked.
- For narrow TS/React changes, start with `bun run test` or the focused `vitest` target.
- For package boundary changes, run `bun run packages:build`, `bun run packages:typecheck`, and `bun run packages:test`.
- For route, auth, database, or cross-cutting changes, run `bun run verify` when time and local services allow.
- E2E setup uses Docker-backed services through `scripts/ci/run-e2e.sh`; it is more expensive than unit and integration tests.

## Files Agents Should Not Edit Manually

- `bun.lock`: update only through Bun dependency operations.
- `next-env.d.ts`: generated by Next.js.
- `db-schema.json`: generated by `bun run db:schema:generate`.
- `drizzle/meta/*.json`: generated by Drizzle migration tooling.
- `packages/*/dist/`, root `dist/`, `.next/`, `out/`, `.generated/`, `coverage/`, `test-results/`, `playwright-report/`, and `public/local-profile-images/`.
- Environment files such as `.env` and `.env.local`; update examples instead when documenting required variables.

## Search And Orientation

- Prefer `rg --files` to list files and `rg "<term>"` to search source.
- Use focused file reads (`sed -n`, `nl -ba`) after search results identify likely owners.
- Useful starting points: `package.json`, `README.md`, `ARCHITECTURE.md`, `docs/platform-layout.md`, `docs/development.md`, `app.config.ts`, and `app.manifest.ts`.

## Docker And Codex/T3 Cleanup

This project uses Docker services for local development or tests: `postgres mailpit minio`.

Standard commands:

- `services:up`: start persistent local development services.
- `services:down`: stop persistent local development services without deleting volumes.
- `services:clean`: remove compose services, orphans, and disposable volumes.
- `services:ps`: show compose service state.
- `test:with-services`: run Docker-dependent tests through trap-based cleanup.
- `check:hygiene`: print git status, upstream/ahead-behind, and Docker state.

Codex/T3 rules:

- Run `git status --short` before editing and before the final response.
- Preserve unrelated dirty files; do not mix them into commits.
- Prefer `test:with-services` for tests that need Docker.
- Do not run raw `docker compose up` for tests unless you also add trap-based cleanup.
- Before finalizing Docker-sensitive work, run `services:ps` or `docker compose ps` and report remaining containers.
- Use `services:down` after dev sessions. Use `services:clean` only when deleting disposable test data is acceptable.
- Format touched files only unless the task explicitly asks for a repo-wide format.
